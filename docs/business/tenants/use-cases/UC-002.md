# UC-002: Provisionar Tenant

**Módulo:** `tenants`, `auth`  
**Classificação de Impacto:** T2  
**Prioridade:** `Must`  
**Esforço relativo:** `L`  
**Dependências:** —  
**Regras de Negócio Acionadas:** RN-001, RN-002, RN-003, RN-004, RN-005, RN-010  
**Ator Principal:** Admin da Plataforma  
**Pré-condições:**
- Admin da Plataforma autenticado com JWT válido contendo `papel = ADMIN_PLATAFORMA`
- Keycloak disponível e acessível pelo backend
- Banco de dados disponível
- O `slug` informado (ou gerado) ainda não está cadastrado na plataforma

**Pós-condições de Sucesso:**
- Tenant criado com `tenant_id` ULID gerado, `status = ATIVO`, registrado no banco de dados
- Usuário admin inicial do tenant criado no Keycloak com atributos `tenant_id` e `papel = ADMIN_TENANT`
- Mapper de claim JWT configurado para este usuário no Keycloak
- Convite de acesso enviado por e-mail ao admin inicial do tenant
- Toast de sucesso exibido; Admin da Plataforma redirecionado para a listagem (UC-001)

**Pós-condições de Falha:**
- Nenhuma operação persistida (rollback completo: Keycloak + banco de dados)
- Mensagem de erro específica ao tipo de falha exibida no formulário

## Análise curta

- **Necessidade:** O Admin da Plataforma precisa criar um novo tenant, vinculando-o ao Keycloak (autenticação) e ao banco de dados compartilhado (isolamento por `tenant_id`), e entregar as credenciais de acesso ao representante do cliente via e-mail.
- **Escopo:** Formulário multi-campo com geração automática de `slug`, suporte a múltiplos telefones, operações atômicas no Keycloak e banco de dados, e envio de convite por e-mail. Não inclui gerenciamento de membros adicionais (módulo RM-F03).
- **Ambiguidades:** QA-02 (comportamento de rollback/compensação em falha parcial Keycloak vs. banco) e QA-03 (desambiguação de `slug` em caso de colisão) ainda estão abertas — registradas abaixo.
- **RNs relevantes:** RN-001 (geração de ULID imutável), RN-002 (unicidade e imutabilidade do `slug`), RN-003 (geração automática do `slug`), RN-004 (JWT com `tenant_id` + `papel`), RN-005 (`tenant_id` como discriminador em todas as tabelas), RN-010 (papéis válidos).

## Fluxos Alternativos

**FA-01: Slug gerado automaticamente** — Bifurca no Passo 3  
À medida que o Admin da Plataforma digita o `nome`, o sistema atualiza o campo `slug` em tempo real aplicando a regra de normalização (RN-003). O campo `slug` permanece editável antes de `confirmar`; após salvar, torna-se imutável (RN-002). Se o Admin editar manualmente o `slug`, o sistema cessa a geração automática para aquela sessão do formulário.

**FA-02: Adição de múltiplos telefones** — Bifurca no Passo 5  
O Admin da Plataforma clica em "+ Adicionar telefone". O sistema exibe um novo campo de telefone com seletor de código de país. O Admin pode adicionar N telefones. Ao clicar no ícone de remoção ao lado de um telefone, o campo é removido do formulário. Os telefones são opcionais; é permitido salvar sem nenhum telefone informado.

## Fluxos de Exceção

**FE-01: Slug duplicado** — Bifurca no Passo 7  
**Gatilho:** O `slug` informado ou gerado já existe na plataforma (violação de RN-002).  
**RN Violada:** RN-002  
**Resposta do Sistema:** Exibe erro inline no campo `slug` com a mensagem _"Este slug já está em uso. Edite para continuar."_. O formulário permanece aberto; o botão de confirmação fica desabilitado até que o conflito seja resolvido.  
> ⚠ **QA-03 em aberto:** Comportamento de desambiguação automática (ex.: sufixo numérico `acme-corp-2`) não está definido. Até resolução, o sistema apenas bloqueia e solicita edição manual.

**FE-02: E-mail inválido** — Bifurca no Passo 7  
**Gatilho:** O campo `email_admin` não é um endereço válido conforme RFC 5322.  
**RN Violada:** — (validação de formato)  
**Resposta do Sistema:** Exibe erro inline no campo `email_admin` com a mensagem _"E-mail inválido."_. O formulário permanece aberto.

**FE-03: Falha no Keycloak** — Bifurca no Passo 10  
**Gatilho:** O Keycloak retorna erro ao criar o usuário ou ao configurar o mapper de claim JWT.  
**RN Violada:** RN-004  
**Resposta do Sistema:** O sistema executa rollback da inserção no banco de dados (Passo 9). Exibe mensagem de erro _"Falha ao configurar autenticação. Nenhuma alteração foi salva. Tente novamente."_. O formulário permanece aberto com os dados preenchidos.  
> ⚠ **QA-02 em aberto:** O mecanismo exato de rollback/compensação (saga, 2-phase commit, idempotência) não está definido.

**FE-04: Falha na persistência no banco de dados** — Bifurca no Passo 9  
**Gatilho:** O banco de dados retorna erro ao persistir o tenant.  
**RN Violada:** RN-005  
**Resposta do Sistema:** Nenhuma operação no Keycloak é iniciada (RN-005 e rollback antecipado). Exibe mensagem de erro _"Falha ao salvar o tenant. Tente novamente."_. O formulário permanece aberto.

**FE-05: Falha no envio do convite por e-mail** — Bifurca no Passo 11  
**Gatilho:** O serviço de e-mail retorna erro ou não responde no timeout configurado.  
**RN Violada:** — (falha de integração assíncrona)  
**Resposta do Sistema:** O tenant e o usuário Keycloak permanecem criados (operações dos Passos 9 e 10 são preservadas). Exibe toast de aviso _"Tenant provisionado, mas o convite não pôde ser enviado. Reenvie manualmente na tela de detalhe."_

---

## Invariantes

- O `tenant_id` gerado como ULID é imutável após a criação e propagado como claim no JWT de todo usuário daquele tenant (RN-001, RN-004).
- O `slug` é imutável após a confirmação do formulário (RN-002).
- O banco de dados deve ter o `tenant_id` como discriminador indexado em todas as tabelas do tenant (RN-005).
- Nenhum token JWT pode ser emitido sem os claims `tenant_id` e `papel` presentes (RN-004).

---

## Ambiguidades

- **QA-02** — O comportamento de rollback/compensação em caso de falha parcial (Keycloak criado, banco falhou — ou vice-versa) ainda não está definido. `confidence_score ≈ 0.70`. A solução adotada na implementação deve ser formalizada antes do desenvolvimento (ADR recomendado).
- **QA-03** — A regra de desambiguação de `slug` em caso de colisão (sufixo numérico, rejeição com edição manual, ou outro) não está definida pela RN-003. `confidence_score ≈ 0.60`. Necessário esclarecimento antes do desenvolvimento da FE-01.

## Critérios de Aceitação (Given/When/Then)

**Cenário 1 — Caminho Feliz:**
```gherkin
Given que o Admin da Plataforma está autenticado com papel = ADMIN_PLATAFORMA
  And o Keycloak e o banco de dados estão disponíveis
  And o slug "acme-corp" não existe na plataforma
When o Admin da Plataforma preenche o formulário com nome="Acme Corp", slug="acme-corp", email_admin="admin@acme.com", primeiro_nome_admin="João", sobrenome_admin="Silva"
  And clica em "Confirmar e Provisionar"
Then o sistema cria o tenant no banco com status = ATIVO e tenant_id ULID gerado
  And o usuário admin é criado no Keycloak com atributos tenant_id e papel = ADMIN_TENANT
  And o e-mail de convite é enviado para "admin@acme.com"
  And o toast "Tenant 'Acme Corp' provisionado com sucesso" é exibido
  And o Admin da Plataforma é redirecionado para a listagem
```

**Cenário 2 — FA-01 (Geração Automática de Slug):**
```gherkin
Given que o campo slug está vazio
When o Admin da Plataforma digita "Acme Corp Brasil" no campo nome
Then o campo slug é preenchido automaticamente com "acme-corp-brasil"
  And o campo slug permanece editável pelo Admin da Plataforma
```

**Cenário 3 — FE-01 (Slug Duplicado):**
```gherkin
Given que o slug "acme-corp" já existe na plataforma
When o Admin da Plataforma submete o formulário com slug = "acme-corp"
Then o sistema exibe erro inline no campo slug: "Este slug já está em uso. Edite para continuar."
  And o botão "Confirmar e Provisionar" permanece desabilitado
  And nenhuma operação é executada no Keycloak ou banco de dados
```

**Cenário 4 — FE-03 (Falha no Keycloak com Rollback):**
```gherkin
Given que o banco de dados está disponível e o Keycloak retorna erro ao criar o usuário
When o processo de provisionamento atinge o Passo 10
Then o sistema executa rollback da inserção no banco de dados
  And nenhum tenant fica registrado no banco
  And a mensagem "Falha ao configurar autenticação. Nenhuma alteração foi salva." é exibida
```
