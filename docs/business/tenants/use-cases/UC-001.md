# UC-001: Listar Tenants

**Módulo:** `tenants`  
**Classificação de Impacto:** T0  
**Prioridade:** `Must`  
**Esforço relativo:** `S`  
**Dependências:** —  
**Regras de Negócio Acionadas:** RN-010, RN-012  
**Ator Principal:** Admin da Plataforma  
**Pré-condições:**
- Admin da Plataforma autenticado com JWT válido contendo `papel = ADMIN_PLATAFORMA`
- Endpoint de listagem de tenants disponível

**Pós-condições de Sucesso:**
- Lista de tenants exibida com `nome`, `slug`, `status` (badge), `membros_ativos` e ações disponíveis por linha

**Pós-condições de Falha:**
- Banner de erro exibido com opção "Tentar novamente"; lista não é renderizada

---

## Tabela de definições de atributos

| Descrição | Nome | Tipo | Domínio | Obrigatoriedade |
|---|---|---|---|---|
| Identificador único do tenant | tenant_id | ULID | Gerado pelo sistema | Sim |
| Nome comercial do tenant | nome | string (máx. 100 chars) | Texto livre | Sim |
| Identificador URL-friendly do tenant | slug | string (máx. 50 chars) | Lowercase, hífens, único na plataforma | Sim |
| Estado atual do tenant | status | enum | ATIVO, INATIVO | Sim |
| Data e hora de criação do registro | criado_em | datetime (UTC) | Gerado pelo sistema | Sim |
| Quantidade de membros com `status_convite = ATIVO` | membros_ativos | inteiro (≥ 0) | Calculado pelo sistema | Sim |

---

## Análise curta

- **Necessidade:** O Admin da Plataforma precisa de uma visão consolidada de todos os tenants para monitorar seu estado e iniciar ações (provisionar, editar, desativar) sem navegar em múltiplas telas.
- **Escopo:** Listagem paginada (20 registros/página) com busca textual por `nome` ou `slug` e filtro por `status`; não inclui criação, edição ou alteração de status — essas ações disparam outros UCs via botões de ação na linha.
- **Ambiguidades:** Nenhuma aberta; campos exibidos, paginação e comportamento de busca estão definidos no protótipo FINAL.
- **RNs relevantes:** RN-012 define paginação de 20 e que `membros_ativos` conta apenas membros com `status_convite = ATIVO`; RN-010 lista os papéis válidos no contexto da plataforma (informativo para este UC, sem validação ativa).

---

## Fluxo Principal

| Passo | Ator | Ação | Sistema |
|---|---|---|---|
| 1 | Admin da Plataforma | Acessa o módulo de Gestão de Tenants | — |
| 2 | — | — | Emite `GET /tenants?page=1&size=20` com JWT no header `Authorization` |
| 3 | — | — | Retorna lista paginada: `tenant_id`, `nome`, `slug`, `status`, `criado_em`, `membros_ativos` |
| 4 | — | — | Renderiza tabela com colunas: Nome, Slug, Status (badge colorido), Membros Ativos, Ações (ícones Visualizar/Editar e Desativar) |
| 5 | Admin da Plataforma | Clica em controle de paginação para página N | — |
| 6 | — | — | Emite `GET /tenants?page={N}&size=20` e atualiza a tabela com a nova página |

---

## Fluxos Alternativos

**FA-01: Busca por nome ou slug** — Bifurca no Passo 1 (campo de busca disponível acima da tabela)  
O Admin da Plataforma digita um termo no campo de busca. O sistema emite `GET /tenants?q={termo}&page=1&size=20`. O resultado substitui o conteúdo da tabela. Se nenhum resultado for encontrado, exibe estado vazio com mensagem _"Nenhum tenant encontrado para '{termo}'"_ (ver FA-03 para estado vazio geral).

**FA-02: Filtro por status** — Bifurca no Passo 1 (seletor de status disponível)  
O Admin da Plataforma seleciona `ATIVO` ou `INATIVO` no seletor de filtro. O sistema emite `GET /tenants?status={ATIVO|INATIVO}&page=1&size=20` e exibe apenas os tenants com o status selecionado. O filtro pode ser combinado com a busca textual (FA-01).

**FA-03: Estado vazio (0 tenants cadastrados)** — Bifurca no Passo 3  
A resposta retorna lista vazia (total = 0). O sistema exibe ilustração de estado vazio com a mensagem _"Nenhum tenant cadastrado"_ e botão de ação primária "Provisionar Tenant" que navega para o formulário de UC-002.

---

## Fluxos de Exceção

**FE-01: Erro de carregamento** — Bifurca no Passo 2  
**Gatilho:** A API retorna erro HTTP (4xx ou 5xx) ou a requisição excede o tempo limite configurado.  
**RN Violada:** — (falha de infraestrutura; sem violação de regra de negócio)  
**Resposta do Sistema:** Exibe banner de erro com mensagem genérica e botão "Tentar novamente". A tabela permanece vazia. Não há redirecionamento automático.

---

## Invariantes

- O sistema DEVE incluir na contagem `membros_ativos` apenas membros com `status_convite = ATIVO` (RN-012).
- A listagem DEVE ser paginada com no máximo 20 registros por página (RN-012).
- Somente usuários com `papel = ADMIN_PLATAFORMA` no JWT estão autorizados a acessar este UC.

## Critérios de Aceitação (Given/When/Then)

**Cenário 1 — Caminho Feliz:**
```gherkin
Given que o Admin da Plataforma está autenticado com papel = ADMIN_PLATAFORMA
  And existem 3 tenants cadastrados (2 ATIVO, 1 INATIVO)
When o Admin da Plataforma acessa o módulo de Gestão de Tenants
Then o sistema exibe uma tabela com 3 linhas
  And cada linha exibe: Nome, Slug, badge de Status, Membros Ativos e ícones de ação
  And o rodapé de paginação indica "Página 1 de 1"
```

**Cenário 2 — FA-03 (Estado Vazio):**
```gherkin
Given que o Admin da Plataforma está autenticado com papel = ADMIN_PLATAFORMA
  And não existem tenants cadastrados
When o Admin da Plataforma acessa o módulo de Gestão de Tenants
Then o sistema exibe a ilustração de estado vazio
  And a mensagem "Nenhum tenant cadastrado" está visível
  And o botão "Provisionar Tenant" está visível e habilitado
```

**Cenário 3 — FE-01 (Erro de API):**
```gherkin
Given que o Admin da Plataforma está autenticado com papel = ADMIN_PLATAFORMA
  And o endpoint GET /tenants está indisponível
When o Admin da Plataforma acessa o módulo de Gestão de Tenants
Then o sistema exibe um banner de erro
  And o botão "Tentar novamente" está visível
  And a tabela não é renderizada
```

**Cenário 4 — FA-02 (Filtro por Status):**
```gherkin
Given que existem 2 tenants com status ATIVO e 1 tenant com status INATIVO
When o Admin da Plataforma seleciona o filtro "INATIVO"
Then o sistema exibe exatamente 1 tenant na tabela
  And o tenant exibido possui badge de status INATIVO
```
